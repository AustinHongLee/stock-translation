# 14. Opus 驗收回覆：估價防誤導與個股頁語意

驗收對象：第一輪防誤導重整（對齊 `docs/12`、`docs/13`）。
本回覆只做「驗收、挑錯、給下一輪優先級」，不重寫藍圖。
所有指控都標到 `檔案:行號`，方便 Codex 直接施工。

結論先講：方向對了，第三層（估價）已經比多數散戶 App 誠實。但**「誠實」目前主要靠文字註腳在撐，視覺與排序仍在偷偷下價值判斷**。小白看的是顏色、排行、數字大小，不是 disclaimer。下一輪要把誠實從「文字層」下沉到「視覺與資料層」。

## 1. 最危險的 5 個剩餘誤導風險

### 風險 1（最高）：價格位階用「綠色 = 好」把「便宜」偷渡回來
`app/explain/rule_based.py:79-84`：價格在近一年低位 → `tone="positive"`（綠），高位 → `tone="caution"`（紅）。
句子雖然寫了「位階只說明相對位置，不等於便宜或該買」，但**顏色已經先講完相反的話**。小白第一眼看到綠色就理解成「現在是好時機」。這是整個 App 最強的隱性買點暗示，而且它在第二層（白話健檢），比第三層估價更早被看到。
→ 位階卡一律用中性色（灰/藍），紅綠只保留給「漲跌」這種事實，不給「位置」這種判斷。

### 風險 2：雷達中心是「撿便宜排行榜」換了名字
`index.html:119,150` + `app.js:556-576`：主榜叫「低於股利情境排行」，每列紅字「低於情境 −X%」，並且**從第 1 名排到第 20 名**。
問題不在用詞，在「排名」這個動作本身——把全市場依「離某個價格多遠」由近到遠排序，功能上就是買點推薦榜，再多的免責也壓不住。小白會直接從上往下當清單買。
→ 見第 3 節：拿掉名次數字、改成體質分群、把「低於情境」從排序鍵降為其中一個標籤。

### 風險 3：固定殖利率三格（6.25% / 5% / 3.125%）看起來還是便宜／合理／貴的階梯
`app/analyze/valuation.py:80-82` 寫死三個目標殖利率，前端 `app.js:1469-1478` 渲染成三張並排卡。
雖然每張卡標題已改成「要求殖利率 X%」、底下也有「股利情境不是合理價」的註解（`app.js:1656`），但**三張由低價到高價並排，視覺上就是一把尺**。而且 6.25 / 5 / 3.125 是任意整除數，不是這檔自己的歷史，卻顯示到小數點第三位，假精確。
→ 這三格與下面「近 5 年平均股利法」（用個股自己的高/平均/低殖利率）功能重疊且更差，建議直接砍掉固定三格，只留個股自身歷史那一組。

### 風險 4：`便宜價／合理價／昂貴價` 還活在資料層，UI 靠字串硬比對
`valuation.py:80-82,184-187` 仍用 `"便宜價"/"合理價"/"昂貴價"` 當 `label`；前端 `app.js:1607-1609` 靠 `estimatesByLabel.get("便宜價")` 反查再改名顯示。
兩個問題：(a) API JSON 仍對外吐「便宜價」這種字，任何測試、匯出、未來新畫面都會直接看到誤導詞；(b) 一旦後端 label 改字，前端 `.get()` 回傳 undefined，畫面靜默變「--」，是顆未爆彈。誠實詞彙只改在最外層渲染、沒改在資料源頭。
→ 後端 label 改成語意中性的 enum（`high_yield`/`avg_yield`/`low_yield`），前端用 enum 對照，不要用中文字串當 key。

### 風險 5：what-if 的「倍數 +20%」被讀成上檔目標價
`app/analyze/methods.py:302`：PE/PB 三格標籤是 `倍數 -20% / 目前倍數 / 倍數 +20%`，中心固定等於現價。
立意良好（中心＝現價、上下只看敏感度），但**「+20%」對應一個比現價高的數字，小白會直接讀成「可以漲到這裡」**。配色與排版又把它跟「目前倍數」並列，像三檔目標價。
→ 把 ±20% 的語意講白：標題改成「市場若多付 / 少付 20% 本益比」，並在卡上明說「這不是預測，方向兩邊一樣可能」；考慮把中心格視覺放大、上下兩格弱化（dim）。

## 2. 四個案例逐檔批判

### 2330 台積電（被分類為成長股 → PE 敏感度）
- 走向正確：低殖利率 + 高 EPS 成長 → `suitability.py:102-107` 會判成 `growth`，路由到 `pe_band` 主尺、`yield` 進 avoid（`suitability.py:233`）。股利法降權成立。
- 仍會被誤認成合理價的點：PE 三格中心＝現價、`fair_price = estimates[1].price = 現價`（`methods.py:154`），所以「目前 PE」那格與現價差永遠是 0，看起來像「現在剛好合理」。這是巧合造成的安心感，不是判斷。
- 缺口：沒有「目前 PE 在自己歷史的高/低位」。對成長股，老手要的是 PE 河流圖 / 歷史本益比 percentile，而不是現價 ±20%。目前的 band 是假 band（`methods.py:309-319` 直接拿現值乘 0.8 / 1.2）。
- 建議：成長股頁面把第一眼資訊換成「營收動能 + 毛利率趨勢 + EPS 連續性」，PE 敏感度退為輔助，並補一句「半導體有景氣循環，單一 PE 不能定錨」。

### 2542 興富發（營建 → PB／資產角度）
- 走向正確：`construction` 路由到 `pb_band` 主尺、避開 yield 與 pe_single（`suitability.py:235`）。brief 也有「盈餘常受建案認列影響」（`api.py:448-449`）。
- 真正的漏洞：**`methods.py` 的 `PB_BANDS` 根本沒有 `construction` 這個 key**（`methods.py:60-68`），會 fallback 到 `general (1.0,1.8,3.0)`。雖然有現值就用現值 ±20%、fallback 多半用不到，但代表營建股沒有任何「量身的判讀」。
- 營建股最缺的防誤導提示（目前完全沒有）：
  1. **每股淨值會嚴重低估**——土地以歷史成本入帳，PB 1 倍不代表沒有資產增值空間（也可能相反）。只給 PB 倍數、不講 NAV/土地重估，會把人帶錯方向。
  2. **盈餘是「完工入帳」一次性灌進來**——某一季 EPS 暴衝不是常態獲利。需要「合約負債 / 預售已售未認列」當前瞻指標，而不是看當期 EPS。
  3. **高股利可能來自單一建案結案**——`one_off_dividend` 偵測（`suitability.py:377-393`）有抓一次性，但營建股要更強硬地標。
- 建議：新增 `construction` 專屬 PB band + 一張「營建股怎麼看」白話卡（完工入帳、合約負債、土地隱含價值三點），PB 敏感度退為輔助。

### 3576 聯合再生（不估價 → 看營收/毛利/EPS）
- 走向正確：虧損史 → `loss_history` / `turnaround_loss` → `_should_refuse_valuation` 回 true（`methods.py:240-250`），出「目前先不估價」拒絕卡（`app.js:1506-1524`）。
- 「產品沒用」風險最高的就是這檔：拒絕卡只給**泛用文字**「先看營收動能、毛利率趨勢、EPS 是否穩定」（`methods.py:265`），但**沒把這三個數字實際算給他看並下白話結論**。小白看到「不估價 + 一段叫他自己去看的話」，會覺得 App 在踢皮球。
- 資料其實都在（營收面板 `index.html:488`、財報面板 `index.html:531`），只是沒有為「不估價股」做一張替代的迷你體質卡。
- 建議（見第 4 節）：不估價時，自動生成「體質觀察」卡，把 4 個具體訊號 + 一句白話判斷推到使用者面前：營收 YoY 連續幾個月正/負、毛利率走勢、EPS 連虧或轉正、是否站回損益兩平。這才是新手要的「替代資訊」。

### 7722 LINE Pay（新上市 → 不估價）
- 走向正確：`newly_listed`（上市未滿 3 年，`suitability.py:119-120`）→ 拒絕卡，理由「上市時間太短，市場還沒有足夠歷史區間可參考」（`methods.py:255-256`）。對。
- 殘留問題：拒絕卡文字偏「資料不足」框架，容易被讀成「系統爛、資料還沒抓」，而不是「這檔本質上現在就不該估」。新上市股要的是**期待管理**——說清楚「再等幾季財報才會有意義」，而不是讓人覺得過幾天同步就會出價格。
- 另外確認：新上市股若被硬丟進雷達中心的殖利率/陷阱欄要小心，目前 `suitability_state` 有過濾（`app.js:489-490` 文案聲稱已排除不適用個股），這點要在第 3 節驗收清單裡實測。

## 3. 雷達中心下一輪改法

核心病根：**它本質是排行榜**。排行榜＝推薦，任何文字都救不回來。下一輪要把它從「便宜排行」改成「分群觀察」。

1. **拿掉名次數字。** `app.js:567` 的 `screener-rank` 1~20 直接移除。改成不排序的卡片牆或標籤分群。沒有名次，就沒有「從上往下買」。

2. **主榜改名 + 改框架。** 「低於股利情境排行」→「股利情境參考」。把「低於情境 −X%」從**排序鍵**降成**其中一個中性標籤**（灰字），不要紅字、不要由近到遠排。明確寫「低於情境不代表便宜，可能是配息要縮水或基本面變差」。

3. **殖利率陷阱區做強，不要只列。** `app.js:611-628` 目前只標「含一次性高股利 / 殖利率陷阱」。補上「為什麼是陷阱」一句話（配息率 >100%？營收連月衰退？一次性業外？），讓陷阱區變成教學，而不是另一個榜。

4. **首頁四個統計數字降溫。** `index.html:115-119` 的「可估價檔數 / 低於股利情境 / …」——「低於股利情境」這個大數字本身就是買點暗示。改成「可估價 / 需留意 / 資料完整」這種體質導向，不要把「便宜檔數」當成就指標。

5. **加一層「為什麼上榜」。** 每張卡點開前就用一個 icon/標籤說明它是因為「殖利率高」還是「跌很多」上榜——這兩件事完全不同，現在混在同一個視覺權重裡。

驗收標準：一個完全不懂股票的人看雷達中心，**講不出「所以我該買哪一檔」**，只講得出「這些是配息穩的、這些要小心」。做到這個才算過關。

## 4. 個股頁 UI 下一輪改法

1. **把估價拒絕卡升級成「體質觀察卡」（對應 3576/7722）。**
   不估價時，不要只給一段叫人自己看的話。自動算並顯示 4 個訊號 + 一句白話結論：
   - 近 3~6 個月營收 YoY 方向（連續正/負/轉向）
   - 毛利率趨勢（改善 / 惡化 / 持平）
   - EPS 連虧幾季 or 是否轉正
   - 是否站回損益兩平
   一句白話：「目前還在虧損收斂中，重點看毛利率能不能連兩季回升，不是看股價便宜。」

2. **修掉第二層的顏色誤導（對應風險 1）。** 價格位階卡改中性色。紅綠只給「漲跌」這類事實。

3. **修資訊架構分層標籤。** 目前 `index.html:467` 的「【第二層 · 白話健檢（6卡）】」標在營收面板之前，但真正的 6 卡健檢報告在最後面（`index.html:577`），中間還夾著營收、財報兩張表。標籤與內容錯位。另外盤中報價面板（`index.html:423`）落在「第一層」標籤之上，沒被分層。建議重排：報價（事實）→ 第一層 30 秒結論 → 第二層體質 → 第三層估價試算，標籤緊貼對應內容。

4. **what-if 三格視覺弱化（對應風險 5）。** 中心格（＝現價）放大為主，「±20%」兩格 dim 掉並改名「市場多付/少付 20%」，避免被讀成上下檔目標價。

5. **砍固定殖利率三格（對應風險 3）。** `app.js:1469-1478` 那組與「近 5 年平均股利法」重複且更差，直接移除，個股頁只留個股自身歷史那一組，減少「一把尺」的暗示。

6. **小數點清理（假精確）。** 目標殖利率（3.125%）、`X.XX 倍`、反推 EPS/每股淨值都顯示到 2~3 位小數（`app.js:1470,1564,1559`）。估算值一律降到 1 位或整數，並在反推來源（official_pe_implied）那種值旁標「反推估計，僅供參考」。價格已用 `formatApproxPrice` 加「約」是對的，倍數與基數也要比照。

## 一句話總評

第一輪把「便宜/合理/貴」這些字從畫面上拿掉了，這是對的第一步。但**誤導的不是字，是排名、顏色和把任意數字並排成階梯這些動作**。下一輪的主軸只有一句：把誠實從註腳搬進視覺與排序，並且讓「不估價」的股票也給得出新手看得懂的體質結論——這樣老手才會信、新手才不會覺得沒用。
