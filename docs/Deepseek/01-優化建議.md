## 1. 架構層 — 目前做對的事（優先保持）

以下設計決策在審視後確認為**正確且應堅守**：

1. **零外部依賴**（`pyproject.toml` dependencies 為空）：標準庫 HTTP server + sqlite3 完全符合「本地單機優先」定位。這不該被「升級 FastAPI」這類建議動搖。
2. **模組化單體而非微服務**：`app/sync`、`app/store`、`app/analyze`、`app/explain`、`app/portfolio` 各目錄自帶 `__init__.py`，import 路徑乾淨。
3. **`frozen=True, slots=True` dataclass**：所有 domain model 都是不可變 + 記憶體高效，這是正確的資料流設計。
4. **純計算層與 IO 層分離**：`analyze/`、`explain/`、`portfolio/calculator.py` 沒有 IO，只吃資料結構，可獨立測試。
5. **合規紅線在程式碼中有體現**：解說層明確不輸出買賣建議，測試也驗證了這一點。

## 2. 效能優化 — 具體熱點與建議

### 🔴 P0 — `TwseClient.fetch_profile()` 每次都拉全市場清單

```python
# app/sync/twse.py:79-84
def fetch_profile(self, stock_id: str) -> StockProfile | None:
    stock_id = stock_id.strip()
    for profile in self.fetch_listed_profiles():  # ← 下載整個上市清單
        if profile.stock_id == stock_id:
            return profile
    return None
```

**問題：** 每次同步一支股票，都先下載完整 TWSE 上市公司清單（目前約 1,000 筆），只為了找到其中一筆。`fetch_listed_profiles()` 是一個沉重的 HTTP 請求。

**建議：** 
- 方案 A（最小改動）：在 `StockSyncService` 層級緩存 `fetch_listed_profiles()` 結果，同一 service 實例生命週期內只請求一次。
- 方案 B（更徹底）：首次 `init-db` 或背景任務將全市場 profile 同步到 SQLite，之後 `fetch_profile` 直接查本地。
- 對照《04》的設計意圖（外部資料只進 sync → 寫入 SQLite → 之後只讀本地），方案 B 才是正解。

### 🔴 P0 — `build_stock_payload` 對同一 stock 執行 3 次 `get_daily_prices`

```python
# app/web/api.py:126-138
prices = store.get_daily_prices(stock_id, start_date=start_date, end_date=end_date)
if not prices:
    prices = store.get_daily_prices(stock_id, limit=days)  # 第二次
# ...
valuation_prices = store.get_daily_prices(  # 第三次：5 年
    stock_id, start_date=end_date - timedelta(days=HISTORICAL_VALUATION_DAYS), ...
)
```

**建議：** 一次取 `max(days, HISTORICAL_VALUATION_DAYS)` 天，在記憶體中 slice：
```python
all_prices = store.get_daily_prices(stock_id, start_date=end_date - timedelta(days=max(days, HISTORICAL_VALUATION_DAYS)))
chart_prices = [p for p in all_prices if p.date >= end_date - timedelta(days=days)]
valuation_prices = all_prices  # 或依需要再切
```
可減少 2 次 SQLite round-trip。

### 🟡 P1 — `search_profiles` 三欄 LIKE 全表掃描

```sql
WHERE stock_id LIKE ? OR name LIKE ? OR short_name LIKE ?
```

資料量小時無感，但若同步全市場後（1,000+ 筆），每次搜尋掃全表。建議：
- 代號精確匹配優先（`stock_id = ?` 走 PRIMARY KEY），再 fallback LIKE
- 中長期：SQLite FTS5 虛擬表做公司名稱全文檢索

### 🟡 P1 — `POST /api/sync` 同步阻塞 HTTP 回應執行緒

同步 5 年 TWSE 資料需要數十次 HTTP 請求（每月一次 + 間隔 0.2 秒），在請求執行緒內執行會讓 UI 長時間無回應。

**建議：** 回傳 `202 Accepted` + `sync_run_id`，前端輪詢 `/api/sync/status/{id}`。`sync_runs` 表已存在，只需加上 API 端點。

### 🟢 P2 — `app.js` 的 `resize` 事件觸發完整 `loadStock`

視窗縮放不該重新打 API，只需要 `drawChart(state.chartPrices)` + debounce。

## 3. 可靠性與安全

### 🔴 P0 — 補 HTTP 層整合測試

`server.py` 有 356 行、完整 REST 路由（GET/POST/PUT/DELETE）、路徑遍歷防護、輸入驗證，但**零測試**。建議優先覆蓋：

```
GET  /api/search?q=2330          → 200 + results
POST /api/portfolio/transactions → 超賣 → 400 + 白話錯誤
DELETE /api/portfolio/transactions/999 → 404
GET  /static/../secret.txt       → 404（路徑遍歷防護）
POST /api/sync 缺 stock_id       → 400
PUT  /api/portfolio/transactions/999 → 404
```

可以用標準庫 `http.client` 或 `urllib` 實作，不需引入 pytest-httpx。

### 🟡 P1 — SQLite 並發寫入保護

`ThreadingHTTPServer` 多執行緒可能同時寫 SQLite。`sqlite3` 預設 journal_mode=DELETE，並發寫入會拋 `database is locked`。

**建議：** 在 `ensure_schema()` 加入：
```sql
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
```
WAL 模式允許並發讀取 + 單一寫入，對於本地單機應用非常適合。

### 🟡 P1 — 錯誤訊息不該直接回傳 Python exception

```python
# server.py:100
self._send_error(HTTPStatus.INTERNAL_SERVER_ERROR, str(exc))
```

`str(exc)` 可能包含內部路徑、SQL 語句等資訊。建議：
- 500 錯誤回傳固定白話訊息「伺服器暫時無法處理，請稍後再試」
- 把 `str(exc)` 寫到 log 檔案
- 400 錯誤可保留業務語意（如「賣出股數超過持有」）

### 🟡 P1 — `trade_date` 接受未來日期

```python
# server.py:268
trade_date=date.fromisoformat(trade_date_raw),
```

沒有檢查 `trade_date > date.today()`。建議在 `_validate_portfolio_state` 或 `_portfolio_transaction_from_body` 中加入日期合理性檢查。

## 4. 程式碼品質與可維護性

### 🟡 P1 — `sqlite_store.py`（877 行）職責過重

同一個檔案內包含：
- DDL schema（19–169 行）
- 8 個 domain 的 CRUD（193–748 行）
- 8 個 row→dataclass 轉換函數（751–876 行）

**建議拆分（不改變對外介面）：**
```
app/store/
  __init__.py       # 從 sqlite_store 匯出 SQLiteStore
  schema.py         # SCHEMA_SQL + ensure_schema
  sqlite_store.py   # SQLiteStore 類（__init__ + close + context manager）
  queries/
    profiles.py     # upsert/get/search profiles
    prices.py       # upsert/get daily_prices
    dividends.py    # upsert/get dividend_records
    valuations.py   # upsert/get market_valuations
    revenues.py     # upsert/get monthly_revenues
    financials.py   # upsert/get financial_statements
    watchlist.py    # add/remove/list/watchlist
    portfolio.py    # add/update/delete/get portfolio_transactions
    sync_runs.py    # record_sync_run
  serializers.py    # _profile_from_row, _price_from_row, ... + _d, _dt, _date_or_none
```

### 🟡 P1 — `api.py`（698 行）的 JSON 序列化應抽離

`api.py` 同時負責：
- 業務邏輯（`build_stock_payload`、`build_portfolio_payload`）
- JSON 轉換（`profile_to_json`、`price_to_json`、`quote_to_json`…共 15+ 個函數）
- 格式化工具（`format_money_text`、`format_percent_text`、`_midpoint`、`_difference`…）

**建議：** 
- `app/web/serializers.py`：放 `*_to_json` 和格式化工具函數
- `app/web/api.py`：只留 `build_*_payload` 業務組合邏輯
- 格式化工具甚至可以進一步抽到 `app/web/formatting.py`

### 🟡 P1 — `app.js`（約 1,630 行）需要拆分

單檔涵蓋：全域 state、API 呼叫、Dashboard 渲染、Portfolio CRUD、Stock 詳情、Chart 繪製、Glossary 彈窗、格式化函數。

**建議第一階段（不新增建置工具）：**
```
app/ui/static/
  app.js            # init + 事件綁定（< 150 行）
  js/
    api.js          # getJson / postJson / putJson / deleteJson
    state.js        # state 物件定義
    format.js       # formatMoney / formatPercent / toneClass / escapeHtml
    dashboard.js    # 首頁渲染
    portfolio.js    # 持倉 CRUD + 渲染
    stock.js        # 個股詳情
    chart.js        # Canvas 繪圖
    glossary.js     # 名詞彈窗
```

HTML 中用多個 `<script>` 標籤載入即可，不需 bundler。

## 5. 資料層深度優化

### 🟡 P1 — `fetch_daily_prices` 逐月請求 + 固定間隔效率低

```python
# twse.py:97-106
for index, month_start in enumerate(months):
    prices.extend(self.fetch_daily_prices_for_month(stock_id, month_start))
    if index < len(months) - 1 and self.request_interval > 0:
        time.sleep(self.request_interval)
```

5 年 = 60 個月 = 至少 12 秒（60 × 0.2s）純等待 + 網路時間。建議：
- 利用 `STOCK_DAY_ALL`（`fetch_latest_all_prices`）一次拉全市場日線，再過濾股票
- 或用 `concurrent.futures.ThreadPoolExecutor` 並行請求（需注意 TWSE 限速）

### 🟢 P2 — 無增量同步機制

每次 sync 都是全量拉取，沒有 ETag / If-Modified-Since / 最後更新時間檢查。`sync_runs` 表已記錄每次同步時間，可以作為增量同步的基礎。

以下建議**不改變四層架構**，只在現有模組內擴充：

`state.activePayload` 已包含每檔個股的 summary / report / quote。首頁可顯示：

- 離線時：同步按鈕 disable + tooltip「離線模式」

在 stock 詳情頁的走勢圖上疊加一條水平線（持股平均成本），讓使用者直觀看到「目前股價 vs 我的成本」。

## 10. 不建議現在做的事（避免過度設計）

延續《06-風險與過度設計》的精神：

- ❌ 引入 React / Vue / Svelte（vanilla JS 足以支撐 MVP）
- ❌ 換 FastAPI / Flask（標準庫 HTTP server 符合本地單機定位）
- ❌ 引入 ORM（SQLiteStore 手寫 SQL 已足夠清晰）
- ❌ WebSocket 即時推播（與產品「踏實感而非刺激感」定位衝突）
- ❌ Docker / K8s（單機桌面應用不需要容器化）
- ❌ CI/CD pipeline（目前一人開發，本地腳本足夠）
