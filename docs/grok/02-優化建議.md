# 02. 優化建議

> 依影響力 × 實作成本排序。標記：🔴 P0（本週）· 🟡 P1（兩週）· 🟢 P2（一個月）

---

## 1. 架構與模組邊界

### 🔴 P0 — 前端單檔過大（`app.js` 約 1,630 行）

**現況：** 搜尋、持倉、圖表、名詞、格式化全部塞在一個 IIFE 風格的單檔。

**建議拆分：**

```
app/ui/static/
  js/
    api.js          # getJson / postJson / error handling
    state.js        # 全域 state
    dashboard.js
    portfolio.js
    stock.js
    chart.js
    glossary.js
    format.js       # formatMoney / toneClass / escapeHtml
  app.js            # init + 事件綁定（< 150 行）
```

**效益：** 降低 merge 衝突、方便單元測試、符合《04》「UI 不直連外部 API」的分層精神（UI 內也應分層）。

### 🟡 P1 — `api.py` 與 `sqlite_store.py` 職責偏重

| 檔案 | 行數 | 建議 |
|------|------|------|
| `sqlite_store.py` | ~876 | 拆 `schema.py`（DDL）+ `queries/`（依 domain） |
| `api.py` | ~697 | JSON serializer 抽到 `app/web/serializers.py` |

### 🟢 P2 — 引入輕量 API 契約型別

即使不引入 FastAPI，也建議用 `TypedDict` 或 dataclass 描述 `build_stock_payload` 回傳結構，讓 UI 與測試共用同一契約定義。

## 2. 效能優化

### 🔴 P0 — `resize` 觸發完整 `loadStock`

```200:204:app/ui/static/app.js
window.addEventListener("resize", () => {
  if (state.activeStockId) {
    loadStock(state.activeStockId, { quiet: true, keepSheet: true });
  }
});
```

**問題：** 視窗縮放會重新打 `/api/stocks/{id}`，造成不必要網路與 DB 讀取。

**建議：** resize 時只呼叫 `drawChart(state.chartPrices)`，必要時 debounce 150ms。

### 🟡 P1 — `build_stock_payload` 重複查詢

`api.py` 對同一 `stock_id` 可能執行 3 次 `get_daily_prices`（chart 區間、valuation 5 年、fallback limit）。

**建議：**

1. 一次取 5 年資料，在記憶體 slice 給 chart / valuation  
2. 或新增 `store.get_daily_prices_once(stock_id, max_days=1825)` 快取於 request scope

### 🟡 P1 — 同步阻塞 HTTP 請求

`POST /api/sync` 在請求執行緒內跑完整 5 年同步（`lookback_days` 預設 1825），UI 會長時間卡住。

**建議：**

- 短期：同步期間回傳 `202 Accepted` + `job_id`，前端輪詢狀態  
- 或至少加 `threading` 背景 job + SQLite `sync_runs` 狀態欄位（表已存在）

### 🟢 P2 — 報價輪詢策略

`scheduleQuoteRefresh` 固定間隔輪詢 MIS。建議：

- 盤中（09:00–13:30）縮短間隔  
- 非交易時段停止輪詢  
- 連續失敗時 exponential backoff

## 3. 可靠性與安全

### 🔴 P0 — 補 HTTP 整合測試

`server.py` 含路由、驗證、靜態檔 path traversal 防護，但零測試。建議覆蓋：

- `GET /api/search?q=2330`
- `POST /api/portfolio/transactions` 超賣 → 400
- `DELETE /api/portfolio/transactions/999` → 404
- `GET /static/../server.py` → 404

### 🟡 P1 — SQLite 並發寫入

`ThreadingHTTPServer` 多執行緒同時寫入 SQLite 可能遇到 `database is locked`。

**建議：**

- `SQLiteStore` 連線加 `timeout=30`  
- 寫入路徑使用 `threading.Lock` 或改 `WAL` journal mode（啟動時 `PRAGMA journal_mode=WAL`）

### 🟡 P1 — 輸入驗證加強

`portfolio` 表單前端有 `min="1"`，但 API 層應明確拒絕：

- `shares <= 0`
- `price < 0`
- 未來日期 `trade_date`

### 🟢 P2 — 錯誤訊息使用者友善化

`server._send_error` 直接回傳 Python exception 字串，可能暴露內部細節。建議區分：

- 使用者訊息（繁中白話）
- 開發者 log（`ui_server.err.log`）

## 4. 資料層

### 現況亮點（保持）

`sqlite_store.py` 已為主要查詢表建立索引：

- `idx_daily_prices_stock_date`
- `idx_dividend_records_stock_year`
- `idx_market_valuations_stock_date`
- `idx_monthly_revenues_stock_month`
- `idx_financial_statements_stock_period`
- `idx_portfolio_transactions_stock_date`

### 🟡 P1 — 搜尋體驗

`search_profiles` 若用 `LIKE '%query%'`，資料量大時會變慢。建議：

- 代號精確匹配優先排序  
- 考慮 SQLite FTS5 虛擬表（公司名稱全文搜尋）

### 🟢 P2 — 資料新鮮度指標

`dataStatus` 目前靜態文案。建議 API 回傳：

```json
{
  "last_sync_at": "2026-06-15T09:12:00",
  "stale_stocks": ["2330"]
}
```

讓首頁顯示「3 檔資料超過 7 天未更新」。

## 6. 產品邏輯優化（非技術但高價值）

### 🟡 P1 — 持倉分頁 Tab 目前是死按鈕

HTML 已有「配置健診 / 配息年曆 / 已實現損益 / 稅務」分頁，但無事件綁定。建議：

- 要嘛隱藏未實作 Tab（避免假功能）  
- 要嘛第一個 Tab 加上 `aria-disabled` + tooltip「即將推出」

### 🟡 P1 — 首頁晨報可規則化（不需 LLM）

`buildDashboardBrief` 已有雛形，可串接：

- 持倉損益加總  
- 自選股漲跌排行  
- 資料缺口警示  

這比等待 AI 更能提升「每天打開」的價值。

### 🟢 P2 — 離線模式提示

本地優先是核心賣點，建議偵測 `navigator.onLine`，離線時：

- 停用同步按鈕  
- 顯示「離線模式：僅顯示本地資料」

## 8. 不建議現在做（避免過度設計）

依《06-風險與過度設計》原則，以下**現階段不建議**：

- 引入 React/Vue 重寫 UI（現有 vanilla 已能支撐 MVP）  
- 換 FastAPI / Electron（標準庫 HTTP server 符合本地單機定位）  
- 微服務拆分（模組化單體已是正確路線）  
- 即時 WebSocket 推播（與「踏實感而非刺激感」設計原則衝突）
