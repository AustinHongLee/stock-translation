# 06. 風險分析與過度設計警示

---

## 1. 風險分析

| # | 風險 | 等級 | 說明 | 對策 |
|---|------|------|------|------|
| R1 | **法規:投顧紅線** | 🔴 高 | 在台灣,對不特定人提供個股買賣「建議」可能落入證券投資顧問業務範圍(需執照)。AI 解說若寫出「建議買進」就有觸法疑慮 | 解說層三鐵則之「AI 不給建議」+ 輸出關鍵字過濾 + 全產品固定免責聲明。**對外發布前,法規範圍請諮詢專業意見並確認最新規定**(本文件非法律建議) |
| R2 | **AI 幻覺講錯數字** | 🔴 高 | LLM 自行編造或算錯數字,使用者據此決策會造成實害並摧毀信任 | 架構級防範:數字一律由分析層計算,AI 只解讀(見《04》)。測試時抽查報告數字與資料庫比對 |
| R3 | **免費資料源不穩** | 🟠 中 | 公開 API 可能限流、改格式、停止服務 | 轉接器隔離(換源只改一個檔案)+ 本地資料庫保留歷史(斷源仍可看舊資料)+ 第 0 天驗證使用條款 |
| R4 | **範圍蔓延** | 🟠 中 | 股票軟體功能想像空間極大,4 週很容易變 4 個月 | 《02》的「MVP 明確不包含」清單 + 每週五可展示日強制砍範圍 |
| R5 | **同事 ≠ 真實散戶** | 🟠 中 | 同事可能客氣、或本身太懂/太不懂股票,回饋失真 | 回饋表問具體行為(哪句看不懂)而非滿意度;優先找「有買股票但看不懂看盤軟體」的同事 |
| R6 | **LLM API 成本與依賴** | 🟡 低 | 每次健檢都呼叫 LLM,使用者變多成本上升;API 中斷產品不能死 | 解說快取(同一股票同一天只生成一次)+ 規則式備援文字。MVP 期間人少,成本可忽略 |
| R7 | **一人開發時程風險** | 🟡 低 | 生病、本業忙碌即停擺 | 模組化讓每個模組都是小的完成單位;每週五都有可展示版本,任何時候中斷都有東西可給 |
| R8 | **資料正確性** | 🟠 中 | 除權息(公司配股配息造成股價跳動)等調整若沒處理,走勢圖會出現假崩跌 | MVP 用還原前價格並在圖上標註除權息日;完整還原價列入第二階段。W1 驗收即包含與證交所網站比對 |

## 2. 哪些設計可能是過度設計(明確不做清單)

| # | 誘人的設計 | 為什麼是過度設計 | 替代做法 |
|---|-----------|----------------|---------|
| O1 | 即時報價、盤中推播 | 即時行情授權貴、技術複雜度倍增;且強化散戶最沒價值的行為(盤中焦慮刷新) | 每日收盤資料。MVP 使用者根本不會發現差異 |
| O2 | 全套技術指標(RSI、MACD、布林通道…) | 目標用戶看不懂,堆指標等於把產品變回看盤軟體 | 4 個白話維度。指標等使用者開口要再加,且加一個解釋一個 |
| O3 | 帳號系統 + 雲端同步 | 單機產品不需要;多了註冊流程反而提高試用門檻,還引入個資保管責任 | 本地儲存。多裝置同步列 F6,等真的有人要 |
| O4 | 微服務 / 訊息佇列 / 容器編排 | 一人開發的單機應用,分散式架構是純成本 | 模組化單體:目錄即模組、介面即契約(《04》),AI Agent 一樣可以局部重做 |
| O5 | 自建 ML 模型預測股價 | 做不到、違反產品定位、踩 R1 法規紅線 | 永久排除 |
| O6 | 多市場(美股、港股)抽象層 | 「以後可能要支援」是典型的過早抽象,先寫兩套轉接器介面卻只有一個市場在用 | 轉接器模式本身已是足夠的擴充點,介面留著、實作只寫台股 |
| O7 | 自製高互動圖表引擎 | 圖表庫是成熟領域,自製是無底洞 | 用現成圖表庫,夠用就好 |
| O8 | 全網新聞爬蟲 | 爬蟲維護成本高、版權有疑慮 | 第二階段再評估,優先用有公開 API 或 RSS 的來源 |
| O9 | 把名詞字典做成 AI 即時生成 | 名詞解釋是固定知識,每次都呼叫 LLM 是浪費錢又慢 | 靜態字典檔,人工審一次,品質還更穩 |
| O10 | MVP 就做精美視覺設計系統 | 4 週內打磨設計系統會吃掉核心功能時間 | 用現成 UI 元件庫的預設樣式,把時間花在報告內容品質 |

> 判斷原則:每當想加一個東西,回到《01》問「它解決哪個編號的痛點?」答不出來就進這張表。
