# 10. 小白產品主與專家驗收施工規格

> 本文件把《09-旗艦前導書》的願景收斂成可施工、可驗收的工作方式。
> 核心前提是：產品主本人可以完全不懂股票；股票正確性必須由系統設計、測試、資料來源與懂股票的人驗收來承擔。

---

## 0. 結論

本產品不能要求產品主判斷「股票分析是否正確」。產品主只負責判斷：

- 小白是否看得懂
- 操作是否直覺
- 畫面是否不嚇人
- 產品是否守住「不報明牌、不預測、不給買賣建議」

股票領域正確性由三層保護：

1. **公式與資料來源寫清楚**
2. **每個計算都有單元測試與範例**
3. **每個階段產出專家驗收清單，交給會股票的人檢查**

換句話說：使用者可以是小白，但產品本身不能小白。

---

## 1. 角色分工

| 角色 | 負責什麼 | 不負責什麼 |
|------|----------|------------|
| 產品主 | 定義產品方向、判斷小白能否看懂、決定功能取捨 | 不判斷財務公式、估價模型、技術指標是否專業正確 |
| Codex / 開發者 | 實作功能、寫公式、寫測試、標資料來源、整理專家驗收包 | 不宣稱投資結論正確、不替使用者做買賣決策 |
| 股票專家 / 朋友 | 驗收資料欄位、公式、語意、是否容易誤導 | 不需要懂程式碼、不負責 UI 工程 |
| 小白試用者 | 回饋「看不看得懂、敢不敢用、哪裡卡住」 | 不負責驗證股票專業內容 |

---

## 2. 每個股票功能的固定交付物

任何新股票功能都必須同時交付 5 件事，不能只交付畫面。

### 2.1 使用者畫面

- 新手第一眼看到一句白話答案
- 原始數字可以展開
- 名詞可以點開解釋
- 固定顯示資料日期與資料來源
- 固定標註非投資建議

### 2.2 計算規格

每個數字都要能回答：

- 這個數字叫什麼？
- 用哪些輸入算出來？
- 公式是什麼？
- 資料來源是哪裡？
- 哪些情況下不能算？
- 這個數字容易被誤解在哪裡？

### 2.3 測試案例

至少包含：

- 正常案例
- 缺資料案例
- 邊界案例
- 買賣或除權息造成的特殊案例
- 與手算結果一致的固定範例

### 2.4 專家驗收清單

給懂股票的人看的驗收表，語氣要像這樣：

- 這個欄位名稱是否正確？
- 這個公式是否符合台股常見理解？
- 這句白話是否會誤導新手？
- 哪些數字需要補充限制條件？
- 是否有不該出現的買賣建議？

### 2.5 小白驗收清單

給產品主或一般試用者看的驗收表：

- 我是否知道這頁在回答什麼問題？
- 我是否知道下一步可以看哪裡？
- 我是否被太多數字嚇到？
- 我是否能分辨「事實」和「建議」？
- 我是否知道這不是明牌？

## 3. 施工紅線

### 3.1 永久不讓產品主背股票判斷責任

如果產品主問「這樣算對嗎？」系統不能回答「你覺得呢？」。

正確處理方式：

1. 列出公式與來源
2. 用測試資料手算一次
3. 標出不確定處
4. 產出給股票專家看的驗收問題

### 3.2 AI 只翻譯，不決策

允許：

- 解釋 ROE 是什麼
- 說明某日成交量比過去 20 日均量高
- 把持倉集中度翻成人話

禁止：

- 建議買進
- 建議賣出
- 預測會漲會跌
- 宣稱合理價就是應買價格
- 把新聞事件說成股價漲跌的確定原因

### 3.3 專業內容必須可追溯

畫面上看到的專業結論，必須能追到：

- 原始資料
- 資料日期
- 計算公式
- 白話解釋規則

## 4. 旗艦第一刀：持倉翻譯機 MVP

《09》提出「個人投資作業系統」，但第一刀不能做完整作業系統。第一刀只做一個閉環：

> 我輸入自己買了什麼，系統告訴我現在帳面發生什麼事，並且用小白能懂的話講清楚。

### 4.1 第一刀目標

讓使用者可以手動新增交易，系統用本地日線收盤價推導目前持倉與帳面損益。

### 4.2 第一刀範圍

- 新增交易紀錄
- 編輯 / 刪除交易紀錄
- 由交易紀錄推導目前持倉
- 計算平均成本
- 計算成本總額
- 讀取最近收盤價
- 計算市值
- 計算未實現損益
- 計算未實現報酬率
- 顯示一句白話解讀
- 顯示資料來源與資料日期
- 產出專家驗收表

### 4.3 第一刀不做

- 不做券商串接
- 不做自動下單
- 不做即時盤中持倉跳動
- 不做 XIRR
- 不做稅務
- 不做多帳戶
- 不做 CSV 匯入
- 不做 AI 助理
- 不做新聞歸因
- 不做除權息完整成本調整

> 除權息與配股會影響成本與股數，第一刀先標為「待專家驗收後進第二刀」，避免第一版把帳算錯。

---

## 5. 持倉 MVP 的資料模型

### 5.1 交易紀錄

`portfolio_transactions`

| 欄位 | 說明 |
|------|------|
| id | 本地交易 ID |
| stock_id | 股票代號 |
| trade_date | 交易日期 |
| side | buy 或 sell |
| shares | 股數 |
| price | 每股成交價 |
| fee | 手續費 |
| tax | 證交稅，買進通常為 0 |
| note | 交易筆記，例如「朋友推薦，想觀察 AI 題材」 |
| created_at | 建立時間 |
| updated_at | 更新時間 |

### 5.2 持倉不另存

目前持倉一律由交易紀錄推導，不另存一張持倉表。

理由：

- 避免交易紀錄與持倉數字對不起來
- 方便測試
- 方便日後重算
- 方便專家驗收

## 6. 持倉 MVP 的計算規格

### 6.1 買進

```
買進總成本 = 股數 * 成交價 + 手續費
買進後股數 = 原股數 + 買進股數
買進後成本總額 = 原成本總額 + 買進總成本
平均成本 = 買進後成本總額 / 買進後股數
```

### 6.2 賣出

第一刀支援賣出，但用移動平均成本法計算已實現損益。

```
賣出收入 = 股數 * 成交價 - 手續費 - 證交稅
賣出對應成本 = 賣出股數 * 賣出前平均成本
已實現損益 = 賣出收入 - 賣出對應成本
賣出後股數 = 原股數 - 賣出股數
賣出後成本總額 = 原成本總額 - 賣出對應成本
```

限制：

- 不允許賣出超過目前持股
- 賣出後股數為 0 時，成本總額歸 0
- 第一刀不處理融資、融券、零股特殊費率

### 6.3 未實現損益

```
市值 = 目前股數 * 最近收盤價
未實現損益 = 市值 - 成本總額
未實現報酬率 = 未實現損益 / 成本總額
```

限制：

- 如果沒有最近收盤價，不能計算市值與未實現損益
- 如果成本總額為 0，不計算報酬率
- 最近收盤價不是即時價格

### 6.4 白話解讀範例

正報酬：

> 你目前持有 2330 台積電 2,000 股，平均成本約 905.5 元。以最近收盤 1,058 元估算，帳面約賺 305,000 元，報酬率約 +16.8%。這只是帳面損益，還沒賣出前不算真正落袋。

負報酬：

> 你目前持有 3008 大立光 100 股，平均成本約 2,680 元。以最近收盤 2,475 元估算，帳面約虧 20,500 元，報酬率約 -7.6%。這代表目前市價低於你的成本，但不等於系統建議你賣出或攤平。

缺資料：

> 這檔股票已有交易紀錄，但本地還沒有最近收盤價，因此暫時不能計算市值與帳面損益。請先同步日線資料。

## 7. 專家驗收表：持倉 MVP

請股票專家或熟悉台股的人檢查以下項目。

### 7.1 交易與成本

- [ ] 買進總成本是否應包含手續費？
- [ ] 賣出收入是否應扣除手續費與證交稅？
- [ ] 第一版使用移動平均成本法是否可接受？
- [ ] UI 是否需要標示「成本算法：移動平均」？
- [ ] 零股、融資、融券是否可先排除？

### 7.2 損益

- [ ] 未實現損益公式是否符合一般券商 App 的直覺？
- [ ] 報酬率分母用成本總額是否合理？
- [ ] 最近收盤價作為市值估算是否需要更明顯標示？
- [ ] 買賣手續費、證交稅的處理是否容易被誤解？

### 7.3 文案

- [ ] 「帳面賺 / 帳面虧」是否比「賺 / 賠」更安全？
- [ ] 是否有任何句子像買賣建議？
- [ ] 是否需要補「不含股利」或「尚未計入配息」？
- [ ] 新手看完是否可能誤以為系統在叫他加碼或停損？

### 7.4 第一版缺口

- [ ] 不處理除權息是否可以接受？
- [ ] 不處理配股成本調整是否可以接受？
- [ ] 不處理 XIRR 是否可以接受？
- [ ] 不處理稅務是否可以接受？
- [ ] 哪個缺口最該排到第二刀？

---

## 8. 小白驗收表：持倉 MVP

請完全不懂股票的人檢查以下項目。

- [ ] 我知道這頁是在看「我目前持有什麼」
- [ ] 我知道「平均成本」大概是我買進的平均價格
- [ ] 我知道「帳面損益」還不是真正落袋
- [ ] 我知道價格是最近收盤，不是即時價格
- [ ] 我知道系統沒有叫我買或賣
- [ ] 我看得懂目前是賺、虧，還是缺資料
- [ ] 我知道要去哪裡新增交易
- [ ] 我知道要去哪裡看原始交易紀錄

## 9. Codex 施工拆分

### Phase M1-1：純計算與測試

目標：

- 建立交易模型
- 建立持倉推導純函數
- 建立未實現損益計算
- 寫固定範例測試

驗收：

- 買進一筆可推導股數、成本、平均成本
- 多筆買進可算移動平均成本
- 賣出一筆可算已實現損益
- 賣出超過持股會失敗
- 缺價格時不算市值

不做：

- UI
- SQLite migration
- CSV
- XIRR

### Phase M1-2：本地資料庫

目標：

- 新增交易紀錄資料表
- 實作新增、編輯、刪除、查詢
- API 回傳持倉推導結果

驗收：

- 重開 App 後交易仍存在
- 刪除交易後持倉會重算
- API payload 可追溯每個持倉來自哪些交易

不做：

- 加密
- 匯入匯出
- 多帳戶

### Phase M1-3：持倉頁 UI

目標：

- 新增「持倉」入口
- 顯示總覽 KPI
- 顯示持股表
- 顯示交易紀錄
- 支援新增交易表單
- 顯示白話解讀

驗收：

- 使用者可手動新增 2330 買進紀錄
- 畫面能顯示股數、平均成本、市值、帳面損益
- 每個數字旁有資料日期或公式說明入口
- 新手模式不被表格淹沒
- 進階區可看到原始交易紀錄

不做：

- AI 助理
- 進階圖表
- 配息年曆

### Phase M1-4：專家驗收包

目標：

- 產出一頁專家驗收報告
- 列出公式、測試範例、畫面截圖、已知限制

驗收：

- 股票專家能在 30 分鐘內完成檢查
- 每個「不確定」都有下一步處理
- 驗收意見能轉成後續 issue / phase

不做：

- 新功能

---

## 10. 第一刀完成定義

當以下條件都成立，才算持倉 MVP 第一刀完成：

- 使用者能新增一筆買進交易
- 系統能推導目前持倉
- 系統能用最近收盤價算帳面損益
- 系統能用白話說明帳面狀態
- 系統不輸出任何買賣建議
- 測試涵蓋成本與損益公式
- 有專家驗收清單
- 有小白驗收清單
- 所有限制都顯示在畫面或文件裡
